## High

### 1. [program.md](/Users/auchan/projects/autoresearch-mlx/program.md#L95)
- Severity: high
- Category: Holistic
- Description: The autonomous workflow instructs agents to `LOOP FOREVER`, avoid asking for human confirmation, and discard losing experiments with `git reset --hard <previous kept commit>`. In a shared or dirty worktree, that can destroy unrelated local changes and conflicts with the repository's otherwise cautious editing model. The lack of a bounded stop condition also makes unattended execution hard to control safely.
- Suggested fix: Replace the hard-reset step with a non-destructive workflow on an isolated branch or worktree, restore only experiment-owned files, and add explicit stop conditions plus a required human opt-in for unattended loops.

### 3. [train.py](/Users/auchan/projects/autoresearch-mlx/train.py#L354)
- Severity: medium
- Category: Holistic
- Description: The main training harness is organized as a single import-time script with module-level hyperparameters and side effects that begin immediately at startup. Importing `train.py` seeds MLX, loads the tokenizer, opens dataloaders, constructs the model, and enters the training path, which makes the core logic difficult to unit test, reuse from helper scripts, or modify safely.
- Suggested fix: Refactor the script into a `main()` entrypoint and pure helpers for config construction, model setup, optimizer setup, and the training loop. Move runtime settings into a typed config object with CLI or environment overrides.

### 4. [pyproject.toml](/Users/auchan/projects/autoresearch-mlx/pyproject.toml#L1)
- Severity: medium
- Category: Holistic
- Description: The project declares only runtime dependencies and does not configure tests, linting, or type checking. There is also no `tests/` tree. For a repository where most logic lives in `train.py` and `prepare.py`, that leaves schedule math, mask generation, optimizer behavior, and startup invariants without fast regression coverage.
- Suggested fix: Add a minimal development toolchain such as `pytest`, a linter, and static typing checks, then cover deterministic units like mask construction, LR scheduling, parameter-group assignment, and data/tokenizer preflight checks.

### 5. [README.md](/Users/auchan/projects/autoresearch-mlx/README.md#L9)
- Severity: medium
- Category: Holistic
- Description: The setup documentation lists `uv sync`, `uv run prepare.py`, and `uv run train.py`, but it does not document any environment variables or startup validation contract because there is no explicit startup validation in the codebase. New developers and agents only discover missing tokenizer files, shards, or platform constraints after lower-level failures are raised from the runtime path.
- Suggested fix: Document the required runtime prerequisites in the README and add explicit startup validation in code so missing data, tokenizer artifacts, or unsupported runtime conditions fail fast with actionable guidance.

## Low

### 6. [train.py](/Users/auchan/projects/autoresearch-mlx/train.py#L433)
- Severity: low
- Category: Holistic
- Description: Run observability is mostly print-based, and the durable experiment log is limited to `results.tsv`. The final output captures aggregate metrics, but not a structured record of the exact hyperparameters, seed, throughput, and timing breakdown for each run, which makes later reproduction and debugging harder once many experiments accumulate.
- Suggested fix: Persist a structured per-run artifact such as JSON or JSONL alongside `results.tsv` that records the resolved config, seed, step count, throughput, timing, and output paths for every experiment.
